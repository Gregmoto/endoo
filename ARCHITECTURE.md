
# Endoo SaaS Invoicing Platform — Comprehensive Architecture Design

## 1. Recommended Tech Stack with Justification

### Core Framework

**Next.js 14+ (App Router)**
The App Router's server components eliminate an entire class of data-fetching patterns that previously required client-side waterfalls. For an invoicing system, most views are read-heavy and benefit from server-rendered HTML for PDF generation, SEO-free dashboards, and instant navigation. The colocation of API routes and UI in the same deployment simplifies infrastructure significantly for a small team.

**TypeScript (strict mode)**
Invoicing data has precise shapes — amounts, tax rates, line items, currency codes. Strict TypeScript with `exactOptionalPropertyTypes: true` will surface bugs at compile time that loose typing would push to runtime. Use `zod` for runtime validation at all I/O boundaries (form submissions, API payloads).

**PostgreSQL 16**
Row Level Security is the cornerstone of this architecture. PostgreSQL's RLS is mature, battle-tested, and integrates naturally with Prisma. `pg_audit` extension handles audit logging natively. `uuid-ossp` or `gen_random_uuid()` for primary keys. `pgcrypto` for any at-rest encryption needs.

**Prisma ORM**
Prisma generates TypeScript types directly from the schema, providing an end-to-end typed data access layer. Its migration system (`prisma migrate`) is the right tool for a team that needs reproducible schema evolution. The Prisma Accelerate connection pooler is worth considering at scale given Next.js serverless functions create new connections on every invocation.

**Authentication: NextAuth.js v5 (Auth.js)**
Supports multiple providers out of the box. Its JWT/session strategy maps cleanly to multi-tenant context passing. The callbacks system is where tenant context will be embedded into the JWT, eliminating a round-trip per request.

### Supporting Stack

| Concern | Choice | Justification |
|---|---|---|
| UI Components | shadcn/ui + Tailwind CSS | Unstyled-by-default, fully owned components. No vendor lock-in for a billing product that needs custom PDF layouts |
| PDF Generation | React PDF (`@react-pdf/renderer`) | Server-side, composable, TypeScript-native. Runs in Node.js context |
| Email | Resend + React Email | Developer-friendly, React-based templates, deliverability is handled |
| File Storage | Vercel Blob or Cloudflare R2 | Invoice PDFs and attachments. R2 is cheaper at scale, zero egress fees |
| Background Jobs | Trigger.dev | Typed job definitions. Handles PDF generation queue, recurring invoice generation, payment reminders |
| Payments | Stripe | Subscriptions for agency/customer plans, plus Stripe's invoicing primitives can be used as a reference model |
| Feature Flags | Vercel Flags or Unleash | Per-tenant feature gating |
| Monitoring | Sentry + Axiom | Sentry for errors, Axiom for structured log querying (audit trail queries) |
| Deployment | Vercel | Zero-config Next.js, edge middleware for tenant routing |

## 2. Database Schema

### Design Principles

- All tenant-scoped tables carry an `organization_id` foreign key. This is the single discriminator used in RLS policies.
- `id` columns use `uuid` generated by `gen_random_uuid()` — never auto-increment sequences, which leak row counts across tenants.
- All tables carry `created_at`, `updated_at` timestamps via a shared trigger.
- Soft delete via `deleted_at` on all business entities. Hard deletes only happen through a scheduled purge job after a configurable retention window.
- Monetary values are stored as `bigint` representing the smallest currency unit (cents/öre). Never `decimal` or `float` for money.

### Complete Schema

```sql
-- ============================================================
-- EXTENSIONS
-- ============================================================
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "pg_trgm"; -- for fuzzy search on customer names

-- ============================================================
-- SHARED TIMESTAMP TRIGGER
-- ============================================================
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- ============================================================
-- ORGANIZATIONS (tenants)
-- Every paying account is an Organization.
-- An Agency is an Organization with type = 'agency'.
-- A Customer is an Organization with type = 'customer'.
-- ============================================================
CREATE TABLE organizations (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug            TEXT NOT NULL UNIQUE,            -- URL-safe identifier: endoo.se/app/acme
  name            TEXT NOT NULL,
  type            TEXT NOT NULL CHECK (type IN ('agency', 'customer')),
  plan            TEXT NOT NULL DEFAULT 'free'     -- free | starter | pro | enterprise
                  CHECK (plan IN ('free', 'starter', 'pro', 'enterprise')),
  plan_expires_at TIMESTAMPTZ,
  stripe_customer_id TEXT UNIQUE,
  stripe_subscription_id TEXT UNIQUE,
  logo_url        TEXT,
  primary_color   TEXT,                            -- hex, for branded PDFs
  invoice_prefix  TEXT NOT NULL DEFAULT 'INV',     -- e.g. ACME-001
  invoice_sequence_start INT NOT NULL DEFAULT 1,
  default_currency TEXT NOT NULL DEFAULT 'SEK',
  default_tax_rate NUMERIC(5,4) NOT NULL DEFAULT 0.25,
  default_payment_terms_days INT NOT NULL DEFAULT 30,
  locale          TEXT NOT NULL DEFAULT 'sv-SE',
  timezone        TEXT NOT NULL DEFAULT 'Europe/Stockholm',
  address_line1   TEXT,
  address_line2   TEXT,
  city            TEXT,
  postal_code     TEXT,
  country         TEXT NOT NULL DEFAULT 'SE',
  vat_number      TEXT,
  org_number      TEXT,                            -- Swedish organisationsnummer
  bank_name       TEXT,
  bank_account    TEXT,                            -- stored encrypted
  bankgiro        TEXT,
  plusgiro        TEXT,
  iban            TEXT,
  bic             TEXT,
  is_active       BOOLEAN NOT NULL DEFAULT true,
  metadata        JSONB NOT NULL DEFAULT '{}',
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ
);

CREATE INDEX idx_organizations_slug ON organizations(slug);
CREATE INDEX idx_organizations_type ON organizations(type);
CREATE INDEX idx_organizations_stripe_customer_id ON organizations(stripe_customer_id);

-- ============================================================
-- AGENCY → CLIENT RELATIONSHIPS
-- Agencies manage multiple client organizations.
-- A client can theoretically be managed by multiple agencies
-- (uncommon but the schema handles it).
-- ============================================================
CREATE TABLE agency_client_relationships (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agency_id       UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  client_id       UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  status          TEXT NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active', 'suspended', 'terminated')),
  granted_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  granted_by      UUID,                            -- user_id of who created the link
  revoked_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(agency_id, client_id)
);

CREATE INDEX idx_acr_agency_id ON agency_client_relationships(agency_id);
CREATE INDEX idx_acr_client_id ON agency_client_relationships(client_id);

-- ============================================================
-- USERS (global identity table, no org affiliation here)
-- Auth providers link to this table via user_accounts.
-- ============================================================
CREATE TABLE users (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email           TEXT NOT NULL UNIQUE,
  email_verified  BOOLEAN NOT NULL DEFAULT false,
  email_verified_at TIMESTAMPTZ,
  full_name       TEXT NOT NULL,
  avatar_url      TEXT,
  locale          TEXT NOT NULL DEFAULT 'sv-SE',
  timezone        TEXT NOT NULL DEFAULT 'Europe/Stockholm',
  last_login_at   TIMESTAMPTZ,
  last_login_ip   INET,
  is_platform_admin BOOLEAN NOT NULL DEFAULT false, -- Super Admin flag
  totp_secret     TEXT,                             -- encrypted
  totp_enabled    BOOLEAN NOT NULL DEFAULT false,
  recovery_codes  TEXT[],                           -- hashed
  metadata        JSONB NOT NULL DEFAULT '{}',
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ
);

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_is_platform_admin ON users(is_platform_admin) WHERE is_platform_admin = true;

-- ============================================================
-- OAUTH / CREDENTIAL ACCOUNTS (NextAuth compatible)
-- ============================================================
CREATE TABLE user_accounts (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  type              TEXT NOT NULL,                 -- 'oauth' | 'email' | 'credentials'
  provider          TEXT NOT NULL,                 -- 'google' | 'github' | 'email'
  provider_account_id TEXT NOT NULL,
  refresh_token     TEXT,
  access_token      TEXT,
  expires_at        BIGINT,
  token_type        TEXT,
  scope             TEXT,
  id_token          TEXT,
  session_state     TEXT,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(provider, provider_account_id)
);

CREATE INDEX idx_user_accounts_user_id ON user_accounts(user_id);

-- ============================================================
-- ORGANIZATION MEMBERS (user ↔ org many-to-many)
-- A user can belong to multiple organizations.
-- role is scoped per membership, not per user globally.
-- ============================================================
CREATE TABLE organization_members (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role            TEXT NOT NULL CHECK (role IN ('owner', 'admin', 'member', 'viewer')),
  invited_by      UUID REFERENCES users(id),
  invited_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  accepted_at     TIMESTAMPTZ,
  is_primary      BOOLEAN NOT NULL DEFAULT false,  -- marks the user's "home" org
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ,
  UNIQUE(organization_id, user_id)
);

CREATE INDEX idx_org_members_org_id ON organization_members(organization_id);
CREATE INDEX idx_org_members_user_id ON organization_members(user_id);
CREATE INDEX idx_org_members_role ON organization_members(organization_id, role);

-- ============================================================
-- INVITATIONS (pending membership invites)
-- ============================================================
CREATE TABLE invitations (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  email           TEXT NOT NULL,
  role            TEXT NOT NULL CHECK (role IN ('owner', 'admin', 'member', 'viewer')),
  token           TEXT NOT NULL UNIQUE DEFAULT encode(gen_random_bytes(32), 'hex'),
  invited_by      UUID NOT NULL REFERENCES users(id),
  expires_at      TIMESTAMPTZ NOT NULL DEFAULT NOW() + INTERVAL '7 days',
  accepted_at     TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(organization_id, email)
);

CREATE INDEX idx_invitations_token ON invitations(token);
CREATE INDEX idx_invitations_email ON invitations(email);

-- ============================================================
-- AGENCY STAFF ACCESS GRANTS
-- Controls which agency members can act on behalf of which client.
-- An agency owner implicitly has access to all clients via the
-- agency_client_relationship, but per-staff restrictions are
-- enforced through this table.
-- ============================================================
CREATE TABLE agency_staff_access (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agency_member_id UUID NOT NULL REFERENCES organization_members(id) ON DELETE CASCADE,
  client_id       UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  access_level    TEXT NOT NULL DEFAULT 'full'
                  CHECK (access_level IN ('full', 'read_only', 'invoicing_only')),
  granted_by      UUID NOT NULL REFERENCES users(id),
  granted_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  revoked_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(agency_member_id, client_id)
);

CREATE INDEX idx_asa_agency_member_id ON agency_staff_access(agency_member_id);
CREATE INDEX idx_asa_client_id ON agency_staff_access(client_id);

-- ============================================================
-- SESSIONS (NextAuth sessions table)
-- ============================================================
CREATE TABLE sessions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_token   TEXT NOT NULL UNIQUE,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  active_organization_id UUID REFERENCES organizations(id),  -- currently active tenant
  impersonating_organization_id UUID REFERENCES organizations(id), -- agency acting as client
  expires         TIMESTAMPTZ NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_sessions_session_token ON sessions(session_token);
CREATE INDEX idx_sessions_user_id ON sessions(user_id);

-- ============================================================
-- VERIFICATION TOKENS (NextAuth)
-- ============================================================
CREATE TABLE verification_tokens (
  identifier      TEXT NOT NULL,
  token           TEXT NOT NULL UNIQUE,
  expires         TIMESTAMPTZ NOT NULL,
  PRIMARY KEY(identifier, token)
);

  notes           TEXT,
  tags            TEXT[] NOT NULL DEFAULT '{}',

  metadata        JSONB NOT NULL DEFAULT '{}',

-- ============================================================
-- PRODUCTS / SERVICES (item catalog)
-- ============================================================
CREATE TABLE products (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name            TEXT NOT NULL,
  description     TEXT,
  sku             TEXT,
  unit            TEXT NOT NULL DEFAULT 'piece'
                  CHECK (unit IN ('piece', 'hour', 'day', 'month', 'kg', 'km', 'service')),
  unit_price      BIGINT NOT NULL DEFAULT 0,       -- in smallest currency unit
  currency        TEXT NOT NULL DEFAULT 'SEK',
  tax_rate        NUMERIC(5,4) NOT NULL DEFAULT 0.25,
  is_active       BOOLEAN NOT NULL DEFAULT true,
  metadata        JSONB NOT NULL DEFAULT '{}',

  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ

CREATE INDEX idx_products_org_id ON products(organization_id);
CREATE INDEX idx_products_sku ON products(organization_id, sku);

-- ============================================================
-- INVOICES
-- ============================================================
CREATE TABLE invoices (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE RESTRICT,
  contact_id      UUID REFERENCES contacts(id) ON DELETE SET NULL,
  invoice_number  TEXT NOT NULL,
  status          TEXT NOT NULL DEFAULT 'draft'
                  CHECK (status IN ('draft', 'sent', 'viewed', 'partial', 'paid', 'overdue', 'void', 'uncollectable')),
  type            TEXT NOT NULL DEFAULT 'invoice'
                  CHECK (type IN ('invoice', 'credit_note', 'quote', 'proforma')),
  currency        TEXT NOT NULL DEFAULT 'SEK',
  issue_date      DATE NOT NULL DEFAULT CURRENT_DATE,
  due_date        DATE NOT NULL,
  sent_at         TIMESTAMPTZ,
  viewed_at       TIMESTAMPTZ,
  paid_at         TIMESTAMPTZ,
  voided_at       TIMESTAMPTZ,

  -- Denormalized totals (computed by trigger / application layer)
  subtotal_amount BIGINT NOT NULL DEFAULT 0,       -- sum of line items before tax
  tax_amount      BIGINT NOT NULL DEFAULT 0,
  discount_amount BIGINT NOT NULL DEFAULT 0,
  total_amount    BIGINT NOT NULL DEFAULT 0,
  paid_amount     BIGINT NOT NULL DEFAULT 0,
  balance_due     BIGINT GENERATED ALWAYS AS (total_amount - paid_amount) STORED,

  -- Billing address snapshot (frozen at send time)
  billing_name    TEXT,
  billing_email   TEXT,
  billing_address JSONB,

  notes           TEXT,
  footer_text     TEXT,
  po_number       TEXT,
  reference       TEXT,

  -- For credit notes
  credited_invoice_id UUID REFERENCES invoices(id),

  -- For recurring
  recurring_schedule_id UUID,                      -- FK set after recurring_schedules table

  -- PDF
  pdf_url         TEXT,
  pdf_generated_at TIMESTAMPTZ,

  -- Created by (which user / which session context)
  created_by_user_id UUID REFERENCES users(id),
  created_on_behalf_of_agency_id UUID REFERENCES organizations(id), -- if agency created it

  metadata        JSONB NOT NULL DEFAULT '{}',
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ,

  UNIQUE(organization_id, invoice_number)
);

CREATE INDEX idx_invoices_org_id ON invoices(organization_id);
CREATE INDEX idx_invoices_contact_id ON invoices(contact_id);
CREATE INDEX idx_invoices_status ON invoices(organization_id, status);
CREATE INDEX idx_invoices_due_date ON invoices(organization_id, due_date) WHERE deleted_at IS NULL;
CREATE INDEX idx_invoices_issued_at ON invoices(organization_id, issue_date DESC);

-- ============================================================
-- INVOICE LINE ITEMS
-- ============================================================
CREATE TABLE invoice_line_items (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  invoice_id      UUID NOT NULL REFERENCES invoices(id) ON DELETE CASCADE,
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  product_id      UUID REFERENCES products(id) ON DELETE SET NULL,
  sort_order      INT NOT NULL DEFAULT 0,
  description     TEXT NOT NULL,
  quantity        NUMERIC(12,4) NOT NULL DEFAULT 1,
  unit            TEXT NOT NULL DEFAULT 'piece',
  unit_price      BIGINT NOT NULL,
  tax_rate        NUMERIC(5,4) NOT NULL DEFAULT 0.25,
  discount_rate   NUMERIC(5,4) NOT NULL DEFAULT 0,
  line_total      BIGINT NOT NULL,                 -- quantity * unit_price * (1 - discount_rate)
  tax_amount      BIGINT NOT NULL,                 -- line_total * tax_rate
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_line_items_invoice_id ON invoice_line_items(invoice_id);
CREATE INDEX idx_line_items_org_id ON invoice_line_items(organization_id);

  stripe_payment_intent_id TEXT,
  created_by_user_id UUID REFERENCES users(id),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_payments_org_id ON payments(organization_id);

-- ============================================================
-- AUDIT LOG
-- Append-only. No updates, no deletes. Protected by RLS.
-- ============================================================
CREATE TABLE audit_logs (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID REFERENCES organizations(id),  -- NULL for platform-level events
  user_id         UUID REFERENCES users(id),
  impersonated_by_user_id UUID REFERENCES users(id),  -- agency staff acting as client

  resource_type   TEXT NOT NULL,                      -- e.g. 'invoice', 'contact'
  resource_id     TEXT,
  old_values      JSONB,
  new_values      JSONB,
  diff            JSONB,                              -- computed diff for updates
  ip_address      INET,
  user_agent      TEXT,
  request_id      TEXT,                               -- correlates to application trace
  metadata        JSONB NOT NULL DEFAULT '{}',
  occurred_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
  -- NO updated_at, NO deleted_at — immutable
);

CREATE INDEX idx_audit_logs_org_id ON audit_logs(organization_id, occurred_at DESC);
CREATE INDEX idx_audit_logs_user_id ON audit_logs(user_id, occurred_at DESC);
CREATE INDEX idx_audit_logs_resource ON audit_logs(organization_id, resource_type, resource_id);
CREATE INDEX idx_audit_logs_action ON audit_logs(organization_id, action, occurred_at DESC);
```

## 3. Multi-Tenant Isolation Strategy (RLS Policies)

### Architecture: Session-Variable Approach

The key challenge with Prisma + RLS is that Prisma does not natively pass per-request PostgreSQL session variables. The pattern to bridge this gap uses a custom Prisma extension that executes `SET LOCAL` commands in a transaction, or a middleware that wraps every query in a transaction with the context set.

### Connection Strategy

```typescript
// lib/db/tenant-client.ts
// This module creates a Prisma client augmented with a tenant context setter.

// The RLS policies read from these session variables:
// current_setting('app.current_organization_id', true)  -- active tenant
// current_setting('app.current_user_id', true)          -- authenticated user
// current_setting('app.is_platform_admin', true)        -- super admin bypass
// current_setting('app.acting_organization_id', true)   -- agency when impersonating
```

Every API route and server action must wrap database operations in a transaction that begins by setting these variables:

```sql
BEGIN;
SET LOCAL app.current_organization_id = '<uuid>';
SET LOCAL app.current_user_id = '<uuid>';
SET LOCAL app.is_platform_admin = 'false';
-- ... your queries ...
COMMIT;
```

### RLS Policy Design

```sql
-- ============================================================
-- ENABLE RLS ON ALL TENANT-SCOPED TABLES
-- ============================================================
ALTER TABLE contacts            ENABLE ROW LEVEL SECURITY;
ALTER TABLE products            ENABLE ROW LEVEL SECURITY;
ALTER TABLE invoices            ENABLE ROW LEVEL SECURITY;
ALTER TABLE invoice_line_items  ENABLE ROW LEVEL SECURITY;
ALTER TABLE payments            ENABLE ROW LEVEL SECURITY;
ALTER TABLE recurring_schedules ENABLE ROW LEVEL SECURITY;
ALTER TABLE expenses            ENABLE ROW LEVEL SECURITY;
ALTER TABLE organization_members ENABLE ROW LEVEL SECURITY;
ALTER TABLE audit_logs          ENABLE ROW LEVEL SECURITY;
ALTER TABLE webhook_endpoints   ENABLE ROW LEVEL SECURITY;

-- Force RLS even for the table owner (critical: prevents accidental bypass)
ALTER TABLE contacts            FORCE ROW LEVEL SECURITY;
ALTER TABLE invoices            FORCE ROW LEVEL SECURITY;
ALTER TABLE payments            FORCE ROW LEVEL SECURITY;
-- (apply FORCE to all above tables)

-- ============================================================
-- HELPER FUNCTIONS (inlined for performance, not PL/pgSQL calls)
-- ============================================================

CREATE OR REPLACE FUNCTION current_org_id() RETURNS UUID AS $$
  SELECT nullif(current_setting('app.current_organization_id', true), '')::UUID
$$ LANGUAGE SQL STABLE SECURITY DEFINER;

CREATE OR REPLACE FUNCTION current_user_id() RETURNS UUID AS $$
  SELECT nullif(current_setting('app.current_user_id', true), '')::UUID
$$ LANGUAGE SQL STABLE SECURITY DEFINER;

CREATE OR REPLACE FUNCTION is_platform_admin() RETURNS BOOLEAN AS $$
  SELECT current_setting('app.is_platform_admin', true) = 'true'
$$ LANGUAGE SQL STABLE SECURITY DEFINER;

-- Returns true if the current session's acting org is an agency
-- that has an active relationship with the target org
CREATE OR REPLACE FUNCTION agency_has_access_to(target_org_id UUID) RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM agency_client_relationships acr
    WHERE acr.agency_id = current_org_id()
      AND acr.client_id = target_org_id
      AND acr.status = 'active'
  )
$$ LANGUAGE SQL STABLE SECURITY DEFINER;

-- ============================================================
-- CONTACTS POLICIES
-- ============================================================
CREATE POLICY contacts_isolation ON contacts
  USING (
    is_platform_admin()
    OR organization_id = current_org_id()
    OR agency_has_access_to(organization_id)
  );

-- ============================================================
-- INVOICES POLICIES
-- Read: own org OR platform admin OR agency with active relationship
-- Write: own org only (agencies can write if they have non-read_only access)
-- ============================================================
CREATE POLICY invoices_read ON invoices
  FOR SELECT
  USING (
    is_platform_admin()
    OR organization_id = current_org_id()
    OR agency_has_access_to(organization_id)
  );

CREATE POLICY invoices_write ON invoices
  FOR ALL  -- INSERT, UPDATE, DELETE
  USING (
    is_platform_admin()
    OR organization_id = current_org_id()
    OR (
      agency_has_access_to(organization_id)
      AND EXISTS (
        SELECT 1
        FROM agency_staff_access asa
        JOIN organization_members om ON om.id = asa.agency_member_id
        WHERE om.organization_id = current_org_id()
          AND om.user_id = current_user_id()
          AND asa.client_id = organization_id
          AND asa.access_level != 'read_only'
          AND asa.revoked_at IS NULL
      )
    )
  )
  WITH CHECK (
    is_platform_admin()
    OR organization_id = current_org_id()
    OR agency_has_access_to(organization_id)
  );

-- ============================================================
-- ORGANIZATION MEMBERS POLICIES
-- Users can see their own memberships.
-- Admins/owners can see all members of their org.
-- ============================================================
CREATE POLICY org_members_read ON organization_members
  FOR SELECT
  USING (
    is_platform_admin()
    OR user_id = current_user_id()
    OR (
      organization_id = current_org_id()
      AND EXISTS (
        SELECT 1 FROM organization_members my_mem
        WHERE my_mem.organization_id = current_org_id()
          AND my_mem.user_id = current_user_id()
          AND my_mem.role IN ('owner', 'admin')
          AND my_mem.deleted_at IS NULL
      )
    )
  );

-- ============================================================
-- AUDIT LOGS POLICIES
-- Append-only: INSERT allowed for the app user, no UPDATE/DELETE.
-- Read: own org (admin+), platform admin.
-- ============================================================
CREATE POLICY audit_logs_insert ON audit_logs
  FOR INSERT
  WITH CHECK (true);  -- Application enforces correct org_id before inserting

CREATE POLICY audit_logs_read ON audit_logs
  FOR SELECT
  USING (
    is_platform_admin()
    OR (
      organization_id = current_org_id()
      AND EXISTS (
        SELECT 1 FROM organization_members mem
        WHERE mem.organization_id = current_org_id()
          AND mem.user_id = current_user_id()
          AND mem.role IN ('owner', 'admin')
          AND mem.deleted_at IS NULL
      )
    )
  );

-- Explicitly prohibit UPDATE and DELETE on audit_logs at DB level
CREATE POLICY audit_logs_no_update ON audit_logs
  FOR UPDATE USING (false);

CREATE POLICY audit_logs_no_delete ON audit_logs
  FOR DELETE USING (false);
```

### The Agency Context-Switching Pattern

When an agency staff member switches to act on behalf of a client, the session must be updated and the RLS variables adjusted:

```sql
-- When agency user "switches into" a client context:
SET LOCAL app.current_organization_id = '<client_org_id>';
SET LOCAL app.acting_organization_id  = '<agency_org_id>';
SET LOCAL app.current_user_id         = '<agency_user_id>';
```

This means `current_org_id()` returns the CLIENT's org, so all queries naturally return only that client's data. The `acting_organization_id` is preserved in the session for audit trail purposes.

## 4. Role and Permission Model

### Two-Layer Model

Layer 1 is **membership roles** (stored in `organization_members.role`), which define what a user can do within their own organization. Layer 2 is **agency access levels** (stored in `agency_staff_access.access_level`), which defines what an agency staff member can do inside a client account.

### Membership Roles

| Permission | viewer | member | admin | owner |
|---|---|---|---|---|
| View invoices | yes | yes | yes | yes |
| Create/edit drafts | no | yes | yes | yes |
| Send invoices | no | yes | yes | yes |
| Void/delete invoices | no | no | yes | yes |
| Manage contacts | no | yes | yes | yes |
| View reports | yes | yes | yes | yes |
| Invite members | no | no | yes | yes |
| Remove members | no | no | yes | yes |
| Change member roles | no | no | admin can't promote to owner | yes |
| Manage billing/plan | no | no | no | yes |
| Delete organization | no | no | no | yes |
| Manage webhooks | no | no | yes | yes |
| Access audit logs | no | no | yes | yes |

### Agency Access Levels

| Permission | read_only | invoicing_only | full |
|---|---|---|---|
| View client data | yes | yes | yes |
| Create/send invoices | no | yes | yes |
| Manage contacts | no | yes | yes |
| Manage members | no | no | yes |
| View audit logs | no | no | yes |
| Change billing | no | no | no (always client owner) |

### Super Admin (Platform Admin)

`users.is_platform_admin = true` bypasses all RLS policies and grants access to everything. This flag is set only via direct database access or a protected internal API behind a second authentication factor. Super admins are never exposed as an organization member of any tenant.

### Permission Checking Pattern in Application Code

```typescript
// lib/auth/permissions.ts — server-side only

type Permission =
  | 'invoice:read' | 'invoice:write' | 'invoice:void' | 'invoice:send'
  | 'contact:read' | 'contact:write'
  | 'member:invite' | 'member:remove'
  | 'org:billing' | 'org:delete'
  | 'audit:read';

const ROLE_PERMISSIONS: Record<string, Permission[]> = {
  viewer:  ['invoice:read', 'contact:read'],
  member:  ['invoice:read', 'invoice:write', 'invoice:send', 'contact:read', 'contact:write'],
  admin:   [...memberPerms, 'invoice:void', 'member:invite', 'member:remove', 'audit:read'],
  owner:   [...adminPerms, 'org:billing', 'org:delete'],
};
```

Every server action and API route calls a `requirePermission(session, 'invoice:send')` guard before executing. This is defence-in-depth on top of RLS — RLS prevents data leakage at the database layer; the permission check prevents unauthorized actions at the application layer.

## 6. Security Considerations

### Data Isolation

**Database layer.** RLS is the last line of defence. Even if application code has a bug that passes the wrong `organization_id`, RLS will silently return zero rows rather than leaking data. `FORCE ROW LEVEL SECURITY` ensures even the database owner role cannot bypass it through a Prisma query.

**Application layer.** Every server action validates:
1. User is authenticated (session check)
2. User has an active membership in the target organization (membership check)
3. User has the required permission for the action (role-permission check)
4. The resource being operated on belongs to the organization in context (ownership check — redundant with RLS but explicit)

**No `organization_id` in URLs.** The active organization is resolved from the session, not from query parameters. A URL like `/invoices/abc-123` is safe even if `abc-123` belongs to another tenant, because the database query will return nothing due to RLS.

### Audit Log Structure

The `audit_logs` table design deserves special attention:

- `organization_id` is nullable to support platform-level events (plan changes, impersonation starts)
- `impersonated_by_user_id` is populated whenever an agency staff member is acting on behalf of a client. Every mutation during an impersonation session records both identities.
- `old_values` and `new_values` store JSONB snapshots. `diff` is computed at insertion time. For financial records (invoices, payments), the full snapshot must be stored — not just the diff.
- The `INSERT`-only RLS policy means even a compromised application credential cannot erase audit history. Purging old audit logs must happen via a separate privileged database role.
- Action names follow the pattern `<resource>.<verb>`: `invoice.sent`, `invoice.voided`, `member.role_changed`, `session.agency_impersonation_started`, `session.agency_impersonation_ended`.

### API Security

**Rate limiting.** Implement at the edge (Vercel middleware or Upstash Redis) per `(user_id, endpoint)` pair. Tighter limits for auth endpoints (5 req/minute for login attempts).

**API keys.** For the public API (Phase 8), generate keys as `endoo_live_<32-bytes-hex>`. Store only the hash (SHA-256) in the database. The raw key is shown to the user exactly once. Keys are scoped to an organization and carry a permission set.

**CSRF protection.** Next.js App Router server actions have CSRF protection built in via origin header validation. For any cookie-based mutation endpoint, validate `Origin` against the expected host.

**Input validation.** All form submissions and API payloads go through `zod` schemas. Schema validation happens in the server action before any database call. Never trust client-computed totals — always recompute `line_total`, `tax_amount`, and `total_amount` server-side.

**Monetary precision.** Reject any request where the client-submitted `total_amount` does not match the server-recomputed total. This prevents manipulation of invoice amounts.

**Secrets management.** Stripe keys, Resend API key, database URL, NextAuth secret are all environment variables. Bank account numbers in `organizations` are encrypted at the application layer using `pgcrypto`'s `pgp_sym_encrypt` before storage. The encryption key lives in the environment, never in the database.

**PDF access control.** Invoice PDFs stored in R2/Blob must not be publicly accessible by URL alone. Use signed URLs with a short expiry (15 minutes) generated server-side. The signing happens only after the request has passed the permission check.

**Agency impersonation safeguards.**

- An `agency_impersonation_started` audit event fires immediately when a switch occurs, recording the agency user, the target client, and the IP address.
- Agency staff cannot elevate their own permissions through impersonation. If a staff member has `read_only` access to a client, the `invoices_write` RLS policy will still block writes.
- The client can see all actions taken on their behalf in their audit log, including the agency identity.
- The platform super admin can revoke any agency-client relationship at any time, immediately terminating all ongoing impersonation sessions (via session invalidation).

### Additional Edge Cases Addressed

**A user belonging to multiple organizations.** The `sessions` table carries `active_organization_id`. The tenant context middleware reads this to set RLS variables. Switching organizations creates a new session record with the new `active_organization_id`. The user's permission level is determined by their `organization_members.role` for the active org — different orgs can have different roles for the same user.

**Permission inheritance.** There is no inheritance. Permissions are flat per role. The `owner` role does not "inherit" from `admin` — each role's permission set is defined entirely in the `ROLE_PERMISSIONS` map. This makes the model auditable and prevents privilege escalation through unexpected inheritance paths.

**Invoice number uniqueness.** The `UNIQUE(organization_id, invoice_number)` constraint in combination with a sequence counter in application code (or a PostgreSQL sequence per org, created dynamically) ensures numbers are gap-free within a tenant and never clash across tenants.

**Soft delete consistency.** Contacts that are soft-deleted must not appear in invoice create/edit dropdowns, but existing invoices that reference the contact must continue to render correctly. The contact's data (name, address) is snapshotted into `billing_address` JSONB on the invoice at send time. After sending, the invoice is self-contained.

**Plan enforcement timing.** Plan limits are checked before the mutation, not after. A "free" org trying to create their 11th invoice this month receives a `402` response with an upgrade prompt. Limit checks run in a transaction with the insert so there is no TOCTOU race.
